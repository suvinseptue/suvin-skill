# 问题树与追问方法论

本文档详细说明如何构建问题树以及按树遍历追问的具体方法。

## 扫描信号词

拿到需求后，**先不问问题**，先扫描所有歧义节点。这些地方一定有歧义：

| 信号 | 例子 | 要挖掘的方向 |
|------|------|--------------|
| 动词模糊 | "管理"、"处理"、"维护" | 具体是增/删/改/查哪些？ |
| 主语缺失 | "可以编辑" | 谁可以？ |
| 条件缺失 | "审核通过后" | 谁审？怎么算通过？ |
| 状态跳跃 | "从A到B" | 中间经过什么？能反向吗？ |
| 数量模糊 | "支持多个" | 有上限吗？ |
| 时间模糊 | "及时通知" | 多久？什么触发？ |

## 问题树输出格式

将问题树写到`docs/question-tree/YYYY-MM-DD-<topic>-tree.md`，格式如下：

```markdown
## 问题树

### A. 功能模块名
- [ ] A1. 待追问的问题？
  - [ ] A1.1 子问题（依赖A1的答案）？
  - [ ] A1.2 另一个子问题？
- [ ] A2. 另一个待追问的问题？

### B. 另一个功能模块
- [ ] B1. ...

### Z. 关联需求（追问过程中发现的潜在需求）
- [ ] Z1. 可能遗漏的需求点？
```

**标记规则：**
- `[ ]` = 待追问
- `[x] → 结论` = 已确认，箭头后写答案
- 缩进层级 = 父子依赖关系
- 大写字母分组 = 功能模块，便于扫描遗漏

## 按树遍历追问

建完问题树后，按树逐节点追问。每问完一个节点：
1. 在树上标记 `[x] → 结论`
2. 检查这个答案有没有带出新的子节点，有则补充到树上
3. 扫描同级节点，确认没有遗漏后再进入下一层

## 追问的六个维度

每一个功能节点，从以下维度判断是否还有歧义：

**① 流程维度**（谁、何时、做什么）
- 谁能触发？什么条件下？触发后系统怎么变？

**② 状态机维度**（有"状态"字样的地方必问）
- 所有状态穷举了吗？每条跳转的触发条件？哪些不可逆？异常状态怎么处理？

**③ 权限维度**（每个操作都要问）
- 谁能看/改/删？角色边界在哪？有所有权限制吗（只能操作自己的）？

**④ 数据维度**（每个字段都要问）
- 必填/选填？格式限制？谁能改？改了有联动吗？历史记录保留吗？

**⑤ 边界异常维度**
- 能撤销吗？并发冲突怎么处理？数据缺失时的降级逻辑？

**⑥ 关联需求维度**
- 这个决定影响哪些其他功能？有没有带出新需求？反向操作有没有考虑？

## 追问优先级

影响面从大到小依次追问，前面的答案会决定后面问题的方向：

```
流程主干 → 权限边界 → 状态与跳转 → 数据字段 → 边界异常 → UI交互细节
```

## 提问的形式规则

- **选择题优先**：给选项，强迫决策，避免「看情况」这种无效答案
- **选项互斥穷举**：不重叠、不遗漏；必要时加「以上都有，不同场景不同规则」作兜底
- **每次只问一个**：多问齐发会导致对方选择性回答，漏掉关键信息
- **先主干后分支**：核心流程清楚后再深入细节
