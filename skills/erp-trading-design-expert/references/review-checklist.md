# ERP 需求方案评审方法论与检查清单

## 评审方法论

### 评审的核心思路

评审 ERP 需求文档不是挑错比赛，而是从专业角度帮助用户降低项目风险、提高方案质量。评审输出应包含三类内容：

1. **肯定合理的设计**：告诉用户哪些设计是正确的，增强信心
2. **指出问题并说明原因**：不能只说"这样不对"，必须解释清楚为什么有问题，实际业务中会遇到什么困境
3. **提供改进方案**：每个问题必须附带至少一个可落地的改进建议

### 评审的六大维度

```
┌─────────────────────────────────────────────────┐
│              ERP 方案评审六维度                    │
├─────────────────────────────────────────────────┤
│  1. 业务完整性  ─  关键业务流程是否全覆盖？        │
│  2. 数据架构    ─  数据模型是否支撑业务和报表？     │
│  3. 多币种合规  ─  外币处理是否符合会计准则？       │
│  4. 流程合理性  ─  审批流、业务流是否高效且可控？   │
│  5. 系统集成    ─  与外部系统的接口是否完整？       │
│  6. 可扩展性    ─  未来业务变化是否容易适应？       │
└─────────────────────────────────────────────────┘
```

### 评审输出的标准格式

对于每个发现的问题，使用以下格式输出：

```
### [问题编号] 问题标题

**严重程度**：🔴 严重 / 🟡 中等 / 🟢 建议优化
**所属维度**：[六维度之一]
**问题描述**：[用用户能理解的语言描述问题是什么]
**实际影响**：[如果不改，实际业务中会遇到什么具体困境]
**改进方案**：[具体的、可落地的改进建议]
**参考依据**：[行业惯例、会计准则、或实际案例]
```

**严重程度定义：**
- 🔴 **严重**：会导致财务数据错误、业务流程无法正常运转、或违反会计准则/法规
- 🟡 **中等**：不影响基本运转但会导致效率低下、数据不完整、或增加人工成本
- 🟢 **建议优化**：当前可用但有更好的实践方案，可以在后续版本迭代

## 检查清单

### 第一维度：业务完整性

#### 1.1 核心业务流程覆盖

- [ ] **采购到付款（P2P）流程**
  - 供应商管理 → 询价/比价 → 采购合同 → 到货验收 → 入库 → 采购发票 → 付款 → 供应商对账
  - 验证点：是否支持暂估入库？发票到票后是否能自动冲回暂估？

- [ ] **销售到收款（O2C）流程**
  - 客户管理 → 报价 → 销售合同 → 发货 → 销售发票 → 收款 → 客户对账
  - 验证点：是否有信用额度管控？超额是否有审批流程？

- [ ] **库存与物流管理**
  - 入库 → 库内管理 → 出库 → 在途管理 → 盘点
  - 验证点：是否区分了在库、在途、保税等不同库存状态？

- [ ] **财务核算流程**
  - 业务凭证生成 → 审核 → 过账 → 期末处理 → 结账 → 报表
  - 验证点：是否做到业财一体？还是业务和财务分别录入？

- [ ] **合同全生命周期管理**
  - 验证点：合同是否是独立的核心对象？还是被简化为订单的附属？

#### 1.2 集装箱/商品贸易特有流程

- [ ] **集装箱管理**（如适用）
  - 箱号管理 → 箱况检验 → 入库/堆场 → 出租/出售 → 修箱 → 报废
  - 验证点：是否有箱号级别的全生命周期追踪？

- [ ] **单证管理**
  - 商业发票 → 装箱单 → 提单 → 原产地证 → 保险单 → 信用证
  - 验证点：单证数据是否从业务系统自动带出，还是需要手工填写？

- [ ] **贸易术语处理**
  - 验证点：系统是否能根据 Incoterms（FOB/CIF/CFR 等）自动划分费用归属？

- [ ] **船期与物流跟踪**
  - 验证点：是否有船期管理功能？能否与物流平台对接自动更新状态？

#### 1.3 容易遗漏的业务场景

- [ ] **退货/退款处理**：采购退货、销售退货的完整流程
- [ ] **索赔处理**：货损、货差的索赔流程
- [ ] **代理业务**：代理采购/代理销售的佣金计算和账务处理
- [ ] **转口贸易**：货物不经过本公司仓库的业务处理
- [ ] **背靠背交易**：采购和销售同步进行的联动处理
- [ ] **样品管理**：样品的出入库和成本分摊
- [ ] **费用分摊**：共享费用（如整船运费）在多个合同间的分摊规则

### 第二维度：数据架构

#### 2.1 主数据管理

- [ ] **客户主数据**
  - 编码规则是否清晰？
  - 是否区分了客户层级（集团→公司→联系人）？
  - 是否包含信用信息（信用额度、账期、付款条件）？
  - 是否支持多地址（办公地址、收货地址、开票地址）？

- [ ] **供应商主数据**
  - 同上客户主数据的检查项
  - 另外检查：是否有供应商评估体系？

- [ ] **商品主数据**
  - 编码规则是否支持分类管理？
  - 计量单位是否支持多单位换算？
  - 是否区分商品类别（集装箱、散货、大宗商品等）？

- [ ] **科目主数据**
  - 科目体系是否完整（资产/负债/所有者权益/收入/成本/费用）？
  - 辅助核算维度是否充分（客户/供应商/部门/项目/币种）？

#### 2.2 数据模型检查

- [ ] **金额字段设计**
  - 是否所有金额字段都使用精确数值类型（DECIMAL），而非浮点型？
  - 是否所有金额字段成对出现（原币 + 本位币）？
  - 单价精度是否足够？

- [ ] **汇率字段设计**
  - 汇率精度是否至少 6 位小数？
  - 是否记录了汇率类型（合同汇率/记账汇率/结算汇率）？

- [ ] **时间字段设计**
  - 是否所有业务单据都有创建时间、修改时间？
  - 涉及跨时区的业务是否用 UTC 存储并记录时区？

- [ ] **编码设计**
  - 业务单据编码是否有清晰的规则（包含日期、类型、序号等）？
  - 是否考虑了并发生成编码的唯一性？

---

### 第三维度：多币种合规

#### 3.1 币种架构

- [ ] 是否明确了本位币（USD）？
- [ ] 是否支持在每笔业务中选择交易币种？
- [ ] 是否支持第三方币种（如供应商用欧元、客户用人民币）？
- [ ] 人民币业务是否有清晰的汇率折算流程？

#### 3.2 汇率管理

- [ ] 是否有汇率表维护功能？
- [ ] 汇率来源是否明确（手工/银行/央行）？
- [ ] 是否区分买入价/卖出价/中间价？
- [ ] 汇率更新频率是否满足业务需求？
- [ ] 是否支持汇率锁定（合同签订时锁定汇率）？

#### 3.3 汇兑损益

- [ ] 是否区分已实现汇兑损益和未实现汇兑损益？
- [ ] 收付款时是否自动计算已实现汇兑损益？
- [ ] 期末是否有外币余额重估机制？
- [ ] 汇兑损益是否能关联到具体交易？

#### 3.4 多币种报表

- [ ] 财务报表是否支持本位币和原币双视角？
- [ ] 应收/应付报表是否按币种分别统计？
- [ ] 资金报表是否按币种分别展示余额？

---

### 第四维度：流程合理性

#### 4.1 审批流程

- [ ] 是否有可配置的审批引擎？
- [ ] 审批规则是否按金额分级？
- [ ] 是否有紧急审批通道？
- [ ] 审批日志是否完整（审批人、时间、意见）？
- [ ] 审批是否支持移动端？

#### 4.2 业务流程衔接

- [ ] 上下游单据是否有清晰的关联关系？（合同→发货→发票→收款）
- [ ] 反向流程是否完整？（退货→退款/红字发票）
- [ ] 单据状态变更是否有明确的前置条件？
- [ ] 是否有超期/逾期的自动提醒机制？

#### 4.3 异常处理

- [ ] 到货数量与合同不符怎么处理？
- [ ] 发票金额与合同不符怎么处理？
- [ ] 客户逾期未付款的处理流程？
- [ ] 供应商交货延迟的处理流程？
- [ ] 汇率大幅波动时的紧急处理流程？

---

### 第五维度：系统集成

#### 5.1 外部系统接口

- [ ] **银行接口**：是否对接网银收付款？银行对账单是否自动导入？
- [ ] **物流接口**：是否对接船公司/物流平台获取物流状态？
- [ ] **海关接口**：是否对接电子口岸/单一窗口？
- [ ] **邮件通知**：关键业务节点是否有邮件/消息通知？
- [ ] **报关系统**：报关数据是否从 ERP 自动导出？

#### 5.2 数据交换

- [ ] 接口数据格式是否标准化（JSON/XML/EDI）？
- [ ] 接口是否有失败重试机制？
- [ ] 接口日志是否完整可追溯？
- [ ] 是否有接口监控和报警？

---

### 第六维度：可扩展性

#### 6.1 业务扩展

- [ ] 新增商品类型是否只需配置不需要开发？
- [ ] 新增币种是否只需添加汇率表？
- [ ] 新增仓库/堆场是否只需主数据配置？
- [ ] 业务规则变更（如审批流程调整）是否可配置？

#### 6.2 报表扩展

- [ ] 是否有自定义报表功能？
- [ ] 是否有数据导出功能（Excel/CSV）？
- [ ] 是否有 BI 对接能力？

#### 6.3 组织架构

- [ ] 是否支持多公司/多实体？
- [ ] 是否支持组织架构调整（部门合并、拆分）？
- [ ] 权限体系是否与组织架构联动？

## 常见需求问题速查表

| 常见问题 | 为什么有问题 | 应该怎么做 |
|---------|------------|-----------|
| 只设计了 USD 和 CNY 两种币种 | 贸易业务可能涉及 EUR、GBP 等，系统不灵活 | 设计为币种表，支持任意币种扩展 |
| 合同没有版本管理 | 合同变更无法追溯，审计风险 | 增加合同版本号和变更记录 |
| 汇率精度只有 2 位小数 | 大额交易中的汇率差异会累积成显著金额差 | 汇率至少 6 位小数 |
| 没有暂估入库流程 | 货到票未到时无法入账，财务数据不及时 | 到货时暂估入库，发票到后冲回 |
| 只有一个"库存"状态 | 在途、保税、冻结库存混在一起，数据不准 | 设计多状态库存管理 |
| 报表只有本位币视角 | 无法了解各币种的实际敞口 | 所有报表支持原币和本位币双视角 |
| 审批流程写死在代码中 | 业务变化时需要开发人员修改 | 使用可配置的审批引擎 |
| 没有期末外币重估 | 资产负债表外币项目失真 | 增加月末/季末自动重估机制 |
| 费用直接按固定比例分摊 | 不同交易的费用结构不同 | 支持多种分摊规则（按重量、按金额、按数量等） |
| 没有信用证管理 | 国际贸易主要结算方式之一 | 增加 L/C 全流程管理 |
| 缺少合规检查 | 可能违反贸易制裁规定 | 集成制裁名单筛查 |
| 没有数据归档策略 | 历史数据越积越多，系统变慢 | 设计数据归档和历史查询方案 |
