# 集装箱与商品贸易公司 ERP 设计原则与经验

## 一、贸易型 ERP 与制造型 ERP 的本质区别

贸易型公司的核心业务是「买进→运输→卖出」，利润来自价差、汇差和服务费，而非生产加工。因此 ERP 设计的核心关注点完全不同：

| 维度 | 制造型 ERP | 贸易型 ERP（集装箱/商品） |
|------|-----------|------------------------|
| 核心对象 | BOM、工单、工序 | 合同、船期、箱号 |
| 利润模型 | 成本归集到产品 | 利润归集到每笔交易（Deal/Contract） |
| 库存管理 | 在制品、成品、原材料 | 在途库存、保税库存、仓库库存 |
| 关键单据 | 工单、领料单、入库单 | 提单(B/L)、信用证(L/C)、商业发票、装箱单 |
| 时间维度 | 生产排程 | 船期跟踪、交货期管理 |
| 风险管理 | 品质风险、交期风险 | 汇率风险、价格波动风险、信用风险 |
| 资金管理 | 成本核算 | 多币种资金管理、结汇/购汇 |

**设计铁律：贸易型 ERP 绝不应该照搬制造型 ERP 的框架再删减模块，而应从交易全生命周期出发正向设计。**

## 二、核心模块设计原则

### 2.1 合同管理（Contract Management）— 系统的心脏

贸易公司的一切业务围绕合同展开，合同是 ERP 的第一公民对象。

**设计原则：**
- **合同 = 利润中心**：每笔合同必须能独立核算收入、成本、费用和利润
- **采购合同与销售合同关联**：支持一对一、一对多、多对一的灵活关联
- **合同状态机**：草稿→审批中→已生效→执行中→已完成→已关闭，每个状态转换有严格的前置条件
- **合同变更管理**：补充协议、价格调整、数量变更必须留痕，保留完整的版本历史
- **合同模板**：按商品类型、客户类型预设模板，减少录入错误

**常见设计错误：**
- ❌ 把合同简化为"订单"处理，丢失了合同条款、付款条件等关键信息
- ❌ 采购和销售合同分开管理，无法做到交易级别的利润追踪
- ❌ 没有合同变更流程，直接修改合同数据导致审计困难

### 2.2 多币种与汇率管理 — 系统的血液

**基本币种架构（以美金为本币）：**

```
交易币种（Transaction Currency）
  → 记录每笔业务的原始币种（USD/CNY/EUR 等）
    → 按约定汇率或即时汇率折算
      → 本位币（Functional Currency = USD）
        → 报表币种（Reporting Currency，可选）
```

**设计原则：**
- **三层汇率体系**：
  - 合同汇率：合同签订时锁定的汇率
  - 记账汇率：入账时使用的汇率（可以是月初汇率、即时汇率等）
  - 结算汇率：实际收付款时的汇率
- **汇兑损益自动计算**：已实现汇兑损益（收付款时）和未实现汇兑损益（期末重估时）
- **期末外币重估**：月末/季末对所有外币余额按期末汇率重估，自动生成汇兑损益凭证
- **汇率来源管理**：支持手工录入和自动获取（央行、银行牌价），区分现汇/现钞买入卖出价
- **人民币业务处理**：对于少量人民币业务，入账时按当日汇率折算为美金本位币

**常见设计错误：**
- ❌ 只记录本位币金额，丢失了原币信息，无法追溯汇率差异
- ❌ 没有区分合同汇率和记账汇率，导致利润核算不准确
- ❌ 缺少期末外币重估机制，资产负债表外币项目金额失真
- ❌ 汇率只精确到小数点后2位（应至少4位，部分货币对需要6位）
- ❌ 人民币业务绕过汇率折算直接入账本位币科目

### 2.3 财务管理（Financial Management）

**设计原则：**
- **双币记账**：每笔凭证同时记录原币和本位币金额
- **科目体系**：参考 US GAAP 或 IFRS 设置科目，按业务性质分设
  - 应收账款按币种和客户辅助核算
  - 应付账款按币种和供应商辅助核算
  - 银行存款按币种和银行账户辅助核算
- **自动凭证生成**：业务单据（采购入库、销售出库、收付款等）自动生成会计凭证
- **利润核算到交易级别**：每笔合同/交易的收入、成本、费用、汇兑损益都能完整追踪
- **资金计划**：按币种做资金收支计划，确保各币种账户有足够余额
- **银行对账**：支持多币种银行对账单导入和自动/手动对账

**应收/应付管理细节：**
- 账龄分析按币种分别统计
- 支持预收/预付款核销
- 信用额度按客户+币种管理
- 逾期提醒和催收流程

### 2.4 采购管理（Purchase Management）

**设计原则：**
- **供应商管理**：供应商分级（战略/核心/一般/备选），定期评估
- **采购询价→比价→合同**：完整的采购流程链
- **价格管理**：支持固定价格、浮动价格（如与期货指数挂钩）、阶梯价格
- **采购入库**：与物流到货关联，支持按箱号/批次入库
- **供应商对账**：多币种对账，支持折扣和扣款

**集装箱采购特殊需求：**
- 箱型管理（20GP、40GP、40HC、45HC、特种箱等）
- 箱龄管理（新箱、次新箱、旧箱的分类定价）
- 箱况检验标准（IICL 标准、CW 标准等）
- 批量采购的分批到货和分批付款

### 2.5 销售管理（Sales Management）

**设计原则：**
- **客户分级管理**：VIP/大客户/一般客户/新客户
- **报价管理**：支持多币种报价，报价有效期管理
- **信用管理**：客户信用额度、账期管控，超限自动预警
- **销售合同→发货→开票→收款**：完整的 O2C（Order to Cash）流程
- **佣金/返利管理**：按销售员、代理商、业务量计算

**商品贸易特殊需求：**
- 背靠背交易（Back-to-Back）：采购合同和销售合同同时签订
- 转口贸易（Entrepôt Trade）：货物不经过本公司仓库
- 代理贸易（Agency Trade）：只收取服务费

### 2.6 库存与物流管理（Inventory & Logistics）

**设计原则：**
- **库存状态管理**：在库、在途、在检、冻结、保税
- **多仓库管理**：自有仓库、第三方仓库、保税仓、港口堆场
- **批次管理**：每批货物可追溯到采购合同
- **库存计价**：加权平均法或个别计价法（贸易公司推荐个别计价法，利润归集更精确）

**集装箱物流特殊需求：**
- **箱号跟踪**：每个集装箱有唯一编号，全生命周期追踪
- **船期管理**：船公司、航线、预计到港/离港时间
- **提单管理**：海运提单（B/L）的签发、背书、放货
- **运费管理**：海运费、内陆运费、港杂费、THC 等各项费用分摊
- **堆场管理**：集装箱在堆场的位置、滞期费计算
- **修箱管理**：箱体维修的工单、费用记录

### 2.7 单证管理（Documentation）

**设计原则：**
- **全套单证模板化**：商业发票(Commercial Invoice)、装箱单(Packing List)、提单(B/L)、原产地证(C/O)、保险单等
- **信用证(L/C)管理**：开证、通知、交单、不符点处理、承兑、付款全流程
- **单证一致性校验**：单单一致、单证一致的自动化校验规则
- **电子化存档**：所有单据的电子版本存档，支持快速检索

**常见设计错误：**
- ❌ 单证系统与业务系统割裂，需要重复录入信息
- ❌ 没有单证审核流程，单据错误导致信用证拒付
- ❌ 缺少单证模板管理，每次手工制作效率低且容易出错

### 2.8 风险管理（Risk Management）

**设计原则：**
- **汇率风险**：
  - 敞口统计：按币种统计未结的应收/应付净额
  - 套期保值：远期结售汇、外汇期权的记录和损益追踪
  - VaR（在险价值）分析：评估汇率波动对利润的影响
- **信用风险**：
  - 客户信用评级体系
  - 信用保险管理（如中信保）
  - 应收账款集中度分析
- **价格风险**：
  - 大宗商品价格波动监控
  - 期货/远期合约对冲管理
- **合规风险**：
  - 贸易制裁名单筛查
  - 反洗钱(AML)合规检查

### 2.9 报表与分析（Reporting & BI）

**必备报表：**

| 报表类别 | 具体报表 |
|---------|---------|
| 财务报表 | 资产负债表、利润表、现金流量表（按本位币和原币） |
| 交易报表 | 合同利润明细表、在执合同汇总表、交易毛利分析 |
| 应收应付 | 账龄分析表（按币种）、客户/供应商余额表 |
| 库存报表 | 库存明细表、在途库存表、库存周转分析 |
| 汇率报表 | 外币敞口报表、汇兑损益明细表、期末重估报告 |
| 业务分析 | 按商品/客户/区域/业务员的多维分析 |
| 风险报表 | 信用额度使用率、客户集中度、汇率敏感性分析 |

## 三、关键设计原则

### 3.1 以交易为中心（Deal-Centric）

每笔交易（Deal/Trade）是利润核算的最小单元。从合同签订到最终收款，所有收入、成本、费用都必须能追溯到具体交易。

```
一笔交易的完整成本构成：
├── 商品成本（采购价格 × 数量）
├── 运输成本
│   ├── 海运费
│   ├── 内陆运费
│   ├── 港杂费
│   └── 保险费
├── 仓储成本
│   ├── 仓储费
│   └── 装卸费
├── 财务成本
│   ├── 信用证费用
│   ├── 贴现利息
│   └── 汇兑损益
├── 关税与税费
│   ├── 进口关税
│   └── 增值税（如适用）
└── 其他费用
    ├── 佣金
    ├── 检验费
    └── 代理费
```

### 3.2 业财一体化

业务操作自动触发财务记账，消除手工凭证录入。

**核心映射关系：**
- 采购入库 → 暂估入账（借：库存商品，贷：应付账款-暂估）
- 采购发票 → 替换暂估（借：应付账款-暂估，贷：应付账款）
- 销售出库 → 收入确认 + 成本结转
- 收款 → 应收核销 + 汇兑损益
- 付款 → 应付核销 + 汇兑损益
- 期末重估 → 未实现汇兑损益

### 3.3 审批流程可配置

不同金额、不同业务类型的审批路径不同，系统必须支持灵活配置：
- 按金额分级审批（例：< $10万直属领导审批，$10-50万部门经理，> $50万总经理）
- 紧急采购快速审批通道
- 超信用额度的销售需要财务总监审批

### 3.4 数据权限与安全

- 按部门/业务组隔离数据（销售员只能看到自己的客户和交易）
- 价格信息分级展示（采购价不对销售人员开放）
- 操作日志完整记录（谁、在什么时间、做了什么操作）
- 关键数据变更需要审批

### 3.5 集成架构

贸易型 ERP 通常需要与以下系统集成：
- **银行系统**：收付款通知、银行对账单、网银支付
- **海关系统**：报关单数据交换
- **物流平台**：船期查询、集装箱追踪
- **电子口岸**：外汇核销
- **税务系统**：发票管理
- **邮件系统**：业务提醒通知

## 四、集装箱贸易特有业务场景

### 4.1 集装箱全生命周期管理

```
采购新箱/二手箱
  → 质检验收（IICL/CW 标准）
    → 入库（堆场/仓库）
      → 出租/出售
        → 租赁：租金收取 → 到期还箱 → 检验 → 维修 → 再出租
        → 销售：发货 → 收款 → 过户
```

### 4.2 集装箱租赁业务

如果公司同时经营集装箱租赁：
- 租赁合同管理（长租/短租/单程）
- 租金计算（日租金 × 天数）
- 还箱管理（损坏检验、维修费用追偿）
- 空箱调配（surplus/deficit 管理）

### 4.3 贸易术语（Incoterms）处理

不同贸易术语影响费用归属和风险转移：
- FOB：卖方负责装船前的费用
- CIF：卖方负责运费和保险
- CFR：卖方负责运费
- EXW：买方负责一切费用

系统必须根据贸易术语自动划分费用归属。

### 4.4 多公司/多实体架构

国际贸易公司通常在多个国家/地区设有子公司：
- 关联交易管理和转移定价
- 合并报表消除内部交易
- 各实体独立核算但可合并查看
- 不同实体可能有不同的本位币

## 五、技术架构建议

### 5.1 推荐技术方案

- **微服务架构**：各核心模块独立部署，通过 API 通信
- **事件驱动**：业务事件（如合同签订、货物到港）触发下游流程
- **CQRS 模式**：读写分离，报表查询不影响业务操作性能
- **多租户设计**：如果系统服务多家公司，需要考虑数据隔离

### 5.2 数据模型核心原则

- 所有金额字段使用 DECIMAL 类型，避免浮点精度问题
- 金额字段成对出现：原币金额 + 本位币金额
- 汇率字段至少精确到小数点后 6 位
- 所有业务单据保留创建时间、修改时间、操作人
- 使用软删除（逻辑删除），不做物理删除
- 关键业务表使用乐观锁防止并发冲突

### 5.3 性能考量

- 大量历史交易数据需要归档策略
- 报表查询考虑预计算/物化视图
- 多币种余额查询的性能优化（缓存策略）
- 批量操作（如月末重估）的异步处理

## 六、常见的 ERP 设计反模式

### 反模式 1：过度 ERP 化

**症状**：把所有流程都塞进 ERP，包括即时通讯、项目管理、OA 审批等
**正确做法**：ERP 专注核心业务流程（合同、物流、财务），其他功能通过集成方式对接专业系统

### 反模式 2：照搬行业标准方案

**症状**：直接使用 SAP/Oracle 的标准贸易模块，不做行业化定制
**正确做法**：以实际业务流程为出发点设计，参考但不照搬标准方案

### 反模式 3：忽视数据质量

**症状**：没有主数据管理规范，客户/供应商/商品编码混乱
**正确做法**：在 ERP 设计初期就建立主数据管理规范和治理流程

### 反模式 4：缺乏扩展性设计

**症状**：所有业务规则硬编码，新增商品类型或业务模式需要大量开发
**正确做法**：使用配置化、参数化的方式管理业务规则

### 反模式 5：财务后置

**症状**：业务系统先做完，最后再考虑财务核算
**正确做法**：业财一体化设计，每个业务流程设计时就同步设计财务凭证映射

### 反模式 6：单币种思维

**症状**：系统只用本位币记录，通过备注字段记录原币信息
**正确做法**：从数据模型开始就支持双币记录，所有报表支持原币和本位币视角
