# 顶级交互设计师 Handoff Skill

面向企业内部系统（ERP、管理后台、B 端工具）的交互设计与 Handoff 文档生成工作流。

**受众**：业务负责人、产品经理、全栈开发工程师  
**交付物**：一份整合的 Markdown Handoff 文档  
**格式约束**：不含系统概览，不含流程索引

---

## 前置步骤：需求澄清（设计前必做，不输出给用户）

拿到需求文档后，**先做完整阅读，识别所有不清晰、有歧义或关键信息缺失的地方，再开始任何设计工作。** 不允许自信推断或静默假设。

### 澄清规则

- **一次只问一个问题**，问完等用户回答，再问下一个
- **优先用选择题**：给出 2—4 个选项，覆盖最可能的答案，并保留「其他」兜底
- **选项要有区分度**：每个选项代表不同的设计决策方向，不是同义词
- **问题顺序按影响范围排序**：先问影响整体结构的（如角色定义、状态流转），再问影响单个字段的（如枚举值、默认值）
- **明确说明为什么要问**：一句话说清楚这个问题影响哪里的设计，让用户理解问题的权重

### 澄清问题的格式

```
[一句话说明这个问题影响哪里的设计]

A. [选项 A 及其含义]
B. [选项 B 及其含义]
C. [选项 C 及其含义]
D. 其他（请补充说明）
```

### 什么情况下可以不问

以下情况允许静默处理，但必须在文档末尾「⚠️ 待确认事项」中列出：

- 枚举字段的具体选项值（不影响交互结构，只影响内容）
- 默认值的具体内容（文档已明确标注「待确认」的）
- 纯文案类问题（按钮文字、提示语措辞）

**不允许静默处理的**：状态流转路径、角色权限边界、必填/非必填判断、操作的成功/失败后续行为。

## 第零步：建立心智模型（设计前必做，不输出给用户）

拿到需求文档后，先在脑中完成三张图，再动笔设计：

**① 实体关系**：识别哪些是主数据（供应商、商品、用户），哪些是单据（申请单、记录），哪些是状态。

**② 状态机**：每种单据的完整生命周期，以及每个状态下允许的操作。

**③ 角色×操作矩阵**：谁能在哪个状态下做什么，决定页面的差异化展示和权限边界。

---

## 第一步：将流程翻译为页面清单（不输出给用户）

| 流程节点类型 | 翻译为 |
|------------|--------|
| 操作节点 | 按钮 / 菜单入口 |
| 查看节点 | 列表页或详情页 |
| 判断节点 | 条件渲染 / 弹窗确认 |
| 系统自动处理节点 | 状态变更 + 消息通知，无需专属页面 |

每个页面记录：页面名称、触发入口、承载的主操作、流向的下一页。

---

## 第二步：逐页设计思考顺序（不输出给用户）

每个页面严格按以下顺序思考：

1. **唯一任务**：这个页面只能有一个主任务（找记录 / 理解全貌 / 准确填写）
2. **用户上下文**：用户带着什么状态进入这个页面（从列表点进、从通知跳转、主动新建）
3. **字段分级**：核心字段（影响决策）→ 关联字段（联动带出）→ 辅助字段（备注/附件/历史）
4. **操作按钮**：主操作唯一，危险操作加二次确认，按钮可见性遵循状态机规则
5. **空态与异常态**：空列表、校验失败、权限不足时的展示和文案

---

## 第三步：输出 Handoff 文档

### 文档结构

```
# [模块名] · Handoff 文档

## 一、角色与权限矩阵

## 二、数据字典
### [实体名]
| 字段 | 来源/类型 | 必填 | 规则与反馈 |
### 统计计算逻辑（如有）

## 三、页面交互说明
### 状态流转总览（Mermaid 状态图）
### P1 · [页面名]
### P2 · [页面名]
### ...
### 企微机器人 / 外部入口（如有）
```

---

### 一、角色与权限矩阵

包含以下内容：

- **页面操作矩阵**（角色 × 操作，✓ / ✗ / 条件说明）
- **记录状态与操作权限**（状态 × 可执行/不可执行操作）
- **外部入口操作权限**（如企微机器人等，如有）

**写法原则**：
- 权限矩阵描述业务逻辑边界，不是系统技术拦截规则
- 条件性权限用括号注明条件，不单独拆行

---

### 二、数据字典

**字段表格固定四列：**

| 字段 | 来源/类型 | 必填 | 规则与反馈 |
|------|---------|------|-----------|

**各列说明：**

- **字段**：界面显示名（括号内附英文字段名）
- **来源/类型**：字段的数据来源或交互组件，例如：
  - `系统生成` `用户输入·文本` `用户输入·正整数`
  - `关联[模块名]·搜索选择` `自动填入·当前账号` `自动填入·今日日期`
  - `枚举·下拉单选` `布尔·系统计算`
- **必填**：`是` / `否` / `条件必填（注明条件）`
- **规则与反馈**：该字段的所有逻辑合并在此列，包括：
  - 联动规则（触发条件 → 联动效果 → 消除/重置条件）
  - 校验规则（合法范围、格式要求）
  - 异常提示文案（校验失败时展示的文字）
  - 只读条件（在什么状态下变为只读）
  - 自动填入说明（默认值来源）

**写法原则：**
- 纯展示、无逻辑的字段可省略不写，或只写来源/类型
- 联动规则优先写，这是开发最容易漏的部分
- 枚举值待确认的字段，在规则列标注 `⚠️ 枚举值待确认`

**统计计算逻辑**（如有聚合统计需求，单独一节）：

说明：统计范围、排除规则、前置拦截条件、统计维度与指标。格式为结构化说明，不是字段表格。

---

### 三、页面交互说明

#### 状态流转总览

用 Mermaid stateDiagram-v2 绘制单据状态机，包含：

- 所有状态节点（含中间态，注明中间态不是独立 status 值）
- 所有状态流转路径和触发动作
- 终态标注

#### 各页面说明

每个页面固定结构：

```
### Pn · [页面名]

**入口：** [从哪里跳转进来]
**角色：** [哪些角色可访问]
**主任务：** [这个页面的唯一核心操作]

#### 字段说明
（只写有逻辑的字段，纯展示字段不写）
[字段 | 组件 | 必填 | 说明] 四列表格，或用文字描述与新建页的差异

#### 操作说明
[按钮名 | 触发条件 | 执行动作 | 成功反馈 | 失败反馈] 五列表格

#### 异常与边界
（空状态、校验规则、无权限、已撤销访问限制等）
```

**写法原则：**
- 操作说明中，**成功反馈必须包含**：Toast/弹窗/跳转目标、页面是否刷新
- 按钮可见性规则（哪个状态下显示 / 隐藏 / 置灰）写在操作说明中，或单独写「按钮展示规则」小节
- 撤销弹窗等非独立页面挂载在列表页，作为独立小节 `Pn · [弹窗名]（挂载于 Pm）`
- 编辑页与新建页差异小时，只写差异部分，其余注明「与新建页一致」

---

## 一致性审查清单（输出前自查，不输出给用户）

- [ ] 同一概念在所有页面使用同一字段名
- [ ] 所有枚举字段已标注值或注明待确认
- [ ] 每个操作的成功反馈和失败反馈均已写明
- [ ] 所有联动规则均已在数据字典中列出
- [ ] 已撤销等终态的操作限制已在每个相关页面说明
- [ ] 待确认事项集中在数据字典枚举值待确认项中，不散落各处

---

## 输出注意事项

1. **一份文档**：所有内容输出到单一 Markdown 文件，不拆分
2. **不含系统概览**：不写模块背景、系统目的、模块地图
3. **不含流程索引**：不写需求流程与设计文档的映射关系
4. **受众适配**：业务方看角色矩阵，开发看数据字典和页面交互说明，产品看状态机。写法保持无歧义、可执行，避免设计术语堆砌
5. **待确认项**：集中在数据字典尾部，用 ⚠️ 标注，不散落正文
